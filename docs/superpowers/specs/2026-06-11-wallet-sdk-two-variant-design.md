# Wallet SDK extraction — two-variant comparison

## Problem

All wallet logic — Cashu and Spark send/receive, accounts, transactions, auth, the
background payment state machines — lives inside the React web app
(`apps/web-wallet/app`), entangled with React and TanStack Query. We want the exact
same logic usable from a future **MCP stdio wallet** (bun/node, same Supabase +
Open Secret backend). The SDK (`packages/wallet-sdk`, `@agicash/wallet-sdk`) must
**completely contain the wallet's business logic and background processing** so that
no frontend ever reimplements either, behind a contract that is clean, minimal, and
understandable on its own.

**What the two variants actually contest is caching.** Business logic, the six
background processors, and leader election live in the SDK in *both* variants. The
open question is whether *freshness/caching of hot reads* is the SDK's job (variant
B: the SDK keeps a live view) or the frontend's job (variant A: the SDK is
operations + events, each frontend caches as it sees fit). That axis — not the
business logic — is what we are comparing.

Two contract philosophies survived design review. Instead of deciding on paper, we
build **both, completely**, and decide by using them.

- **Variant A — stateless**: every read is a `Promise`, every state change is an
  event. The SDK holds no cache.
- **Variant B — store-based**: hot reads are live `Store<T>` views the SDK keeps
  fresh; engine is a hidden, patched `@tanstack/query-core`.

Deliverable: **two self-contained PRs**, each implementing the full SDK, the full
web-app migration, and a headless example — identical in scope and behavior,
differing only in contract shape and engine. The decision is made by comparing them
in action (criteria at the end). Variant C (B's contract on a hand-rolled engine)
is deliberately *not* built: if B wins, C remains a later engine-swap inside B's
seams; if A wins, C is moot.

## Goals & constraints (both variants)

- React-agnostic SDK, fully usable headless under bun/node.
- SDK owns **all business logic and background processing** in both variants:
  operations, the six background payment processors, leader election, retry policy,
  serialization, encryption, version handling. (Caching of hot reads is the one
  responsibility that moves between SDK and frontend across the two variants.)
- Web app keeps its UX and routing; its change is a rebind of data access, not a
  redesign.
- The MCP target runs the same full stack (Supabase RLS + realtime, Open Secret).
- No TanStack symbol crosses the SDK boundary in either variant.
- Same DB schema, same RPCs, same protocol behavior — byte-for-byte the flows the
  app runs today.

## Shared foundation (identical in both PRs)

Both PRs begin from the same base commits (see Delivery) so the ~80% of code that
is contract-independent is written once and is line-identical across PRs.

### Entry point & config

```ts
class Sdk {
  static create(config: SdkConfig): Promise<Sdk>;  // private constructor
  // domains: auth, user, accounts, cashu, spark, transactions, contacts,
  //          transfers, scan, rates, background
  on<E extends keyof SdkEventMap>(e: E, cb: (p: SdkEventMap[E]) => void): () => void;
  resync(): Promise<void>;   // coarse, idempotent host hint: run the catch-up now (re-verify subs + refetch state). Web wires it to visibility/online; MCP rarely needs it.
  dispose(): Promise<void>;  // close sockets, stop processors, drop secret refs
}

type SdkConfig = {
  openSecret: { url: string; clientId: string };
  supabase: { url: string; anonKey: string; serviceRoleKey?: string }; // serviceRole ⇒ server mode
  breezApiKey?: string;
  storage: StorageAdapter;   // web = localStorage, node = fs; holds session tokens ONLY
  clientId?: string;         // leader-election instance id; autogenerated if omitted
  domain?: string;           // LN-address domain for contact composition
};

interface StorageAdapter {
  get(key: string): Promise<string | undefined>;
  set(key: string, value: string): Promise<void>;
  remove(key: string): Promise<void>;
}
```

The SDK constructs and owns the Supabase client, Open Secret session, Breez SDK,
and websocket subscription managers. It never touches `window` / `localStorage` /
`document`; all environment persistence goes through `StorageAdapter`.

### Auth & secrets seam

SDK owns the auth state machine end-to-end: Open Secret email/guest sign-in/up,
guest upgrade, password change, JWT → `generateThirdPartyToken()` → Supabase
session, and key derivation (Cashu seed `m/83696968'/39'/0'/12'/0'`, Spark mnemonic
`…/12'/1'`, encryption key `m/10111099'/0'`). OAuth is a host/SDK split:
`auth.beginGoogle(): Promise<{ authUrl }>` / `auth.completeOAuth(params)` — the
redirect itself is host-owned (web-only; MCP uses email/guest).

Secrets: the raw seed is derived on demand and **never** written through
`StorageAdapter` (session tokens only). `dispose()` drops all key material.
Session-expiry handling (guest auto-extend, signOut on user expiry) moves into the
SDK; the `auth:session-expired` event replaces the app's `window.location.reload()`
recovery.

### Background processing & leader election

The six processors (cashu/spark × send/receive × quote/swap) become react-free SDK
classes with **identical state-machine logic in both variants**, driven by:

- mint websocket subscription managers (NUT-17; 10s polling fallback per mint),
- Breez SDK payment events (with listener-before-lookup race handling),
- Supabase realtime changes,
- expiry timers (`setLongTimeout` at quote expiry).

Leader election lives in the SDK: the existing `wallet.task_processing_locks` row
lock via `take_lead` RPC (5s poll, 6s lease). `background.start()` makes the
instance a candidate; the SDK guarantees exactly one active processor per user
across tabs/devices/MCP sessions; `background.stop()` releases; `background.state`
∈ `stopped | starting | follower | leader | stopping`. Crash recovery is unchanged:
on becoming leader, processors load unresolved/pending work from the DB and resume;
single-writer safety during lease handoff continues to rely on
`meltProofsIdempotent`'s deterministic counter and the DB proof-reservation RPCs.

### Serialization lanes & retry policy

Per-entity serialization is wallet correctness and is preserved exactly, including
the two-lane topology per send quote (`initiate-cashu-send-quote-${id}` separate
from `cashu-send-quote-${id}`). How lanes are *implemented* differs per variant
(A: `KeyedQueue`; B: patched query-core `MutationObserver` scopes) behind a shared
internal seam: `runTask(lane, fn, retryPolicy)` is the only dispatch API processor
code may use.

Retry classification is internal SDK policy in both variants, applied inside SDK
methods so every consumer gets it: `ConcurrencyError` → retry, `DomainError` /
`MintOperationError` → never, transient → bounded (3; 5 for subscription setup).

### Change-feed ingestion

One internal module subscribes the per-user realtime channel (`wallet:${userId}`),
decrypts payloads, converts rows → domain entities, stamps `version`, and fans out
to: (1) the public event bus, (2) variant-specific state (B: cache writes; A:
nothing — events only), (3) processors (trigger checks). Reconnect handling
(exponential backoff, online/active resubscribe) ports from
`SupabaseRealtimeManager`. On channel reconnect the SDK emits `connection:state` and
runs **the catch-up** automatically (internal `onConnected`): re-establish mint/Breez
subscriptions and reconcile current state, because realtime events that fired during
the gap were pushed to a dead socket and are lost (not queued). The catch-up does:
B → active-refetch the hot stores (pushes fresh data to subscribers now, not lazy
invalidation); A → reload the processors' work sets from DB and emit
`connection:resync` so the frontend invalidates its own cache. Processors reload
their work sets in both.

`sdk.resync()` is simply the **on-demand trigger of that same catch-up path** — for
when the socket didn't notice it dropped (device merely slept and missed events) or
to catch up on focus faster than the heartbeat would. It is coarse and idempotent
(safe to call spuriously). The SDK cannot observe `window` online/visibility itself,
so the web host wires `resync()` to those events (replacing today's
`refetchOnWindowFocus`); headless hosts rely on channel-level reconnect detection and
rarely call it.

### Error taxonomy

`SdkError` base; `DomainError` (user-facing, never retried), `ConcurrencyError`
(transient, retried), `NotFoundError` move from `features/shared/error.ts` into the
SDK and are exported. Methods reject with these; `send:failed` / `receive:failed`
carry one.

### Event map (variant-specific — follows the caching split)

The event surface is **not uniform**. A *core* set fires in both variants, but
*row-level entity events* exist **only in variant A**. In B the stores are the change
mechanism, so row events would be dead weight — a public API with no consumer that we
would nonetheless have to support. This is a direct consequence of the
caching-responsibility split: an event that exists only to maintain A's frontend cache
has no place in B's contract.

**Core `SdkCoreEventMap` (both variants).** Flow-lifecycle (semantic) + infra.
Lifecycle events fire **once, on a terminal transition** — distinct from row changes:
a send quote goes UNPAID→PENDING→PAID and its row changes three times, but
`send:completed` fires once. All fire on **every instance** (derived from realtime
entity transitions, not from the processing mutation), so a passive tab or a headless
MCP is notified of a flow another instance processed.

```ts
type SdkCoreEventMap = {
  'send:completed':    { protocol: 'cashu' | 'spark'; quoteId: string; transactionId: string; amount: Money };
  'send:failed':       { protocol: 'cashu' | 'spark'; quoteId: string; transactionId?: string; error: SdkError };
  'receive:completed': { protocol: 'cashu' | 'spark'; quoteId: string; transactionId: string; amount: Money };
  'receive:failed':    { protocol: 'cashu' | 'spark'; quoteId: string; error: SdkError };
  'receive:expired':   { protocol: 'cashu' | 'spark'; quoteId: string };
  'auth:signed-in':       { user: User };
  'auth:signed-out':      Record<string, never>;
  'auth:session-expired': Record<string, never>;
  'connection:state':  { state: 'connected' | 'disconnected' };
  'background:state':  { state: BackgroundState };
};
```

Variant A extends this with row-level events (see Variant A); variant B exposes
`SdkCoreEventMap` unchanged.

### Domain surface

Writes, actions, and on-demand reads are **identical in both variants** (always
`Promise`). The variants differ only in the form of the seven hot reads:

| Hot read | A (stateless) | B (store) |
|---|---|---|
| current user | `user.get(): Promise<User \| null>` | `user.current: Store<User \| null>` |
| accounts | `accounts.list(): Promise<Account[]>` | `accounts.all: Store<Account[]>` |
| contacts | `contacts.list(): Promise<Contact[]>` | `contacts.all: Store<Contact[]>` |
| unresolved cashu sends | `cashu.send.listUnresolved(): Promise<…[]>` | `cashu.send.unresolved: Store<…[]>` |
| pending cashu receives | `cashu.receive.listPending(): Promise<…[]>` | `cashu.receive.pending: Store<…[]>` |
| unresolved spark sends | `spark.send.listUnresolved(): Promise<…[]>` | `spark.send.unresolved: Store<…[]>` |
| pending spark receives | `spark.receive.listPending(): Promise<…[]>` | `spark.receive.pending: Store<…[]>` |

Shared method conventions: writes/actions → `Promise<T>`; lookup by id → `Promise<T | null>`
(A: always; B: also sync `accounts.get(id)` off the loaded store); pure derivations
sync; terminal-state wait → `awaitTerminal(id): Promise<T>`; `Account` carries its
balance as a field (computed at read/feed time) so balance display is uniform.

### Server-mode SDK (LN-address routes)

`Sdk.create` with `serviceRoleKey` yields a server-mode instance (no auth session,
no realtime, no background): used by `api.lnurlp.*` / `.well-known.lnurlp.*`
loaders to serve LUD-16/06/21 — an inbound LN-address payment is a receive-quote
creation the SDK already owns. DB row types stay SDK-internal in both variants;
the app sees only domain types.

### Lib extraction (shared base commits)

General-purpose libs become sibling workspace packages consumed by both app and
SDK: `@agicash/money`, `@agicash/bolt11`, `@agicash/ecies` (+ crypto primitives),
`@agicash/utils` (json/zod/type-utils), and the pure-protocol part of cashu
(`@agicash/cashu`: proof/secret/token/payment-request/error-codes/protocol-extensions).
Wallet-runtime cashu (subscription managers, `ExtendedCashuWallet` factory,
blind-signature matching, mint validation) moves into the SDK. The generated
`supabase/database.types.ts` stays at the repo root; the typed wrapper
(`features/agicash-db/database.ts`) moves into the SDK and becomes internal.
`Money` must be extracted (not copied): SDK domain types reference it and duplicate
classes would break `instanceof` in the app.

## Variant A — stateless: operations + events

**Philosophy:** the DB is the only state; the SDK is operations over it plus a
change feed. Nothing is cached; nothing can be stale; there is no engine.

- All reads hit the DB (through SDK repositories) when called.
- Processors are event-triggered and **read the DB on demand**: an external event
  (websocket/Breez/realtime/timer) names an entity → `runTask(lane, …)` →
  re-read entity from DB → check state → call service transition → DB write →
  realtime → entity event re-emitted. On becoming leader / on reconnect:
  `listUnresolved()`/`listPending()` from DB and resume each.
- Version gating is the consumer's concern only where the consumer keeps state
  (the web app's caches); events carry `version` to make it trivial.

**Row-level events (A only) — `SdkEventMapA = SdkCoreEventMap & {…}`.** A's frontend
rebuilds its cache from these, so they mirror today's Supabase change feed 1:1:
explicit `created` / `updated` (not a collapsed upsert), each carrying the full
decrypted entity + `version`. The handler replaces-by-version and decides
add/update/remove-from-list from the entity's state (exactly what today's Cache
classes do).

```ts
'user:updated'                                                 { user: User }
'account:created' | 'account:updated'                          { account: Account }
'transaction:created' | 'transaction:updated'                  { transaction: Transaction }
'contact:created'                                              { contact: Contact }
'contact:deleted'                                              { contactId: string }
'cashu-send-quote:created' | 'cashu-send-quote:updated'        { quote: CashuSendQuote }
'cashu-send-swap:created' | 'cashu-send-swap:updated'          { swap: CashuSendSwap }
'cashu-receive-quote:created' | 'cashu-receive-quote:updated'  { quote: CashuReceiveQuote }
'cashu-receive-swap:created' | 'cashu-receive-swap:updated'    { swap: CashuReceiveSwap }
'spark-send-quote:created' | 'spark-send-quote:updated'        { quote: SparkSendQuote }
'spark-receive-quote:created' | 'spark-receive-quote:updated'  { quote: SparkReceiveQuote }
'connection:resync'                                            Record<string, never>  // A-only: invalidate + refetch
```

**Web integration:** the app **keeps TanStack Query as its wallet cache**. Read
hooks keep their `queryOptions` with `queryFn: () => sdk.…`; the existing Cache
classes and change handlers survive but are rewired to consume SDK entity events
(already decrypted domain objects) instead of raw Supabase payloads; `staleTime`
and focus/reconnect refetch behavior stay as today, with `connection:resync` as an
additional invalidate-all trigger. Deleted from the app: services, repositories,
the six `useProcess*Tasks` hooks, task-processing/leader election, realtime
decryption, encryption, key derivation. Foreground mutations become thin
`useMutation` wrappers whose `mutationFn` calls `sdk.…` (UI status only).

**Headless usage:**

```ts
const sdk = await Sdk.create(config);
await sdk.auth.signIn({ email, password });
await sdk.background.start();
const quote = await sdk.cashu.receive.createLightningQuote({ account, amount });
console.log(quote.paymentRequest);
const paid = await sdk.cashu.receive.awaitTerminal(quote.id);   // resolves when processor completes it
```

**Known costs to observe in comparison:** DB read amplification (every UI render
path and processor trigger is a query); each reactive frontend must own a cache +
event wiring (the web app keeps ~its current cache plumbing forever); two sources
of read truth (app cache vs DB) with freshness managed app-side.

## Variant B — store-based: hidden query-core

**Philosophy:** the SDK owns one live in-memory view of hot state, kept fresh by
the change feed; consumers subscribe or read snapshots. Engine: patched
`@tanstack/query-core@5.90.20`, fully hidden.

**Reactivity primitive:**

```ts
type Store<T> = {
  get(): T | undefined;                          // sync snapshot; undefined = not yet loaded
  subscribe(listener: () => void): () => void;   // fires on every change
  toPromise(): Promise<T>;                       // resolves on first successful load (Suspense seam)
};
```

Snapshot rules: `get()` returns the observer's cached result (referentially stable
between changes — `useSyncExternalStore` requirement); `undefined` strictly means
"not loaded" (legitimately-empty is `[]`/`null`); `toPromise()` uses
`fetchOptimistic` for true first-load semantics.

**Events:** B exposes only `SdkCoreEventMap`. Account / contact / user / quote-set
freshness is delivered by the stores, not events — there are no row-level change
events to subscribe to, because nothing in B's contract needs them.

**Transactions** are Promise-based in **both** variants (`list(cursor)` / `get(id)` /
`countPendingAck()` / `acknowledge()`) — no transaction store, and **no transaction
store is warranted**, because the current wallet does not live-update history either:
verified in `transaction-hooks.ts`, the realtime handlers upsert only the individual
`['transactions', id]` cache and invalidate the unacked count, while the paginated
`all-transactions` list is a stale-while-revalidate snapshot — `staleTime: 0` +
`refetchOnMount: true` (both defaults) mean revisiting the page shows cached pages
instantly and refetches in the background, plus `refetchOnWindowFocus`/`Reconnect:
'always'`.

**This SWR-on-visit behavior is preserved in BOTH variants for free.** The history
list stays an app-side `useInfiniteQuery`; the only change is its `queryFn` calls
`sdk.transactions.list(cursor)` instead of the repository. Every refetch trigger
(`staleTime`/`refetchOnMount`/focus/reconnect) lives in the app's query config, which
does not move into the SDK — the SDK's `list()` is just the fetch underneath. B keeps
this query in its own UI-side QueryClient (transactions are the one read that stays
app-cached in both variants); an in-flight transaction's detail observes liveness via
the pending/unresolved **stores** until the lifecycle event fires. (A gets identical
behavior, and additionally its `transaction:created` / `transaction:updated` row
events feed the individual-transaction cache exactly as today.)

**Engine rules (hard requirements):**

1. **Seams rule (lint-enforced):** `@tanstack/query-core` may be imported only
   inside `packages/wallet-sdk/src/internal/engine/` — the store factory, the
   `runTask` lane dispatcher (patched `MutationObserver` scopes), and the cache
   handle. Domain/processor code never sees a TanStack type, keeping a later
   engine swap a 3-file change.
2. Explicit `defaultOptions`: node's `isServer` flips default retry → 0 and
   gcTime → Infinity; set retry policy and finite gcTime explicitly.
3. Explicit lifecycle: seed hot queries on start (nothing mounts observers
   headless); resync on realtime reconnect (replaces `refetchOnWindowFocus`);
   offline behavior of `runTask` is pinned at the seam (define, don't inherit
   `onlineManager` semantics silently).
4. The query-core patch, exact version pin, and a CI assertion of a single
   resolved copy move into/with the SDK.
5. Secrets are non-dehydratable; `dispose()` clears the cache.

**Web integration:** new `WalletSdkProvider` + adapter hooks written once
(`useStore`, `useStoreSuspense` — throws `store.toPromise()`, preserving the root
`<Suspense>` — and `useStoreSelect(store, fn)` with memoized selection). Deleted
from the app: Cache classes, wallet `queryOptions`, change handlers, processors,
task-processing, encryption/key derivation, and the query-core patch (it now lives
in the SDK). The app keeps its own `QueryClient` strictly for UI-only queries
(feature flags, exchange-rate polling) and thin foreground `useMutation` wrappers
calling `sdk.…`; the SDK's hidden client is never placed in a Provider; the two
clients never share state.

**Headless usage:** identical flow to A; reads are `store.get()` /
`store.toPromise()` instead of Promise calls.

**Known costs to observe in comparison:** patched dependency carried inside the
SDK (upgrade cost, pin + CI guard); headless-defaults rules must hold forever;
debugging crosses the facade into query-core internals; the repo-wide patch covers
both the SDK engine and the app's own UI QueryClient while the app still uses
TanStack (single pinned copy, CI-asserted — versions must stay in lockstep).

## Delivery plan

Two self-contained PRs against `master`, sharing their first commits:

1. **Base commits** (identical ancestry in both branches, not a separate PR):
   lib extraction (`@agicash/money|bolt11|ecies|cashu|utils`), SDK scaffolding,
   `Sdk.create`/config/auth/StorageAdapter, db wrapper move, repositories,
   services, change-feed ingestion, processor state machines behind `runTask`,
   error taxonomy, event bus, server-mode SDK + LN-address route rewire.
2. **Branch `sdkx/stateless`** → PR A: Promise read surface, `KeyedQueue` lane
   dispatcher, DB-on-demand processor reads, web migration (caches/change handlers
   rewired to SDK events), headless example script, tests.
3. **Branch `sdkx/store`** → PR B: `Store<T>` surface, engine module (query-core +
   patch + seams lint), store seeding/resync, web migration (adapter hooks; cache
   plumbing deleted), headless example script, tests.

Branch namespace `sdkx/` avoids collision with the existing `sdk/*` and
`sdk-reactive/*` stacks. A base defect found while building either variant is fixed
in the base lineage and propagated to both branches. Implementation runs in fresh
sessions (one worktree per branch) driven by three implementation plans (base, A,
B) produced via `writing-plans`.

**Prerequisite spike (before base):** Supabase realtime over websockets headless in
bun — both variants depend on it; if it fails, design the polling fallback before
building the change feed. Spark/Breez headless (WASM under bun/node) is validated
during base (needed for MCP later; the web build already runs it browser-side).

**Tests (same suite shape in both PRs):** ported send/receive state-machine tests
against SDK processors; lane-serialization tests (FIFO per lane, initiate-lane
isolation); version-gate ordering (stale event must not regress state);
reconnect-resync; leader election multi-instance (exactly one processor under
concurrent `start()`, lease handoff without double-processing); headless smoke
(example script against local stack). B adds: store first-load semantics,
referential stability, seeding under `isServer`. A adds: app cache/event wiring
tests.

## Evaluation criteria (the comparison)

Judged side by side, same local stack, same flows:

1. **Web integration diff** — size and complexity of the app-side changes per
   domain; how much permanent plumbing the app retains (A: caches + event wiring;
   B: adapter hooks).
2. **Headless ergonomics** — the two example scripts read side by side; what an
   MCP tool implementation would look like in each.
3. **SDK readability** — open `sdk.ts` + one processor and one domain: can a new
   maintainer explain data flow in ten minutes?
4. **Behavior parity** — pay an invoice → balance/tx visible live in web UI;
   receive while a second tab is leader; kill the leader mid-flow → other instance
   completes it; airplane-mode reconnect → resync; duplicate/out-of-order realtime
   events do not regress state.
5. **Runtime characteristics** — DB query counts per flow (A's read
   amplification), startup time to interactive, memory; web bundle delta.
6. **Debuggability** — trace a stuck PENDING quote in each: how many layers to
   look through, what logging exists.
