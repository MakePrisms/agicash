# Smart Source Account Selection for BOLT11 — Design

**Status:** Draft
**Date:** 2026-04-21
**Branch:** `feat/universal-qr-scanner` (builds on the universal QR scanner work)
**Author:** gudnuf (+ Claude)

## Motivation

The universal QR scanner (commit `5d7d62ff`) classifies pasted and camera input and routes BOLT11 invoices to `/send`, but the send flow always initializes with the user's default account. When a scanned invoice was generated by a mint the user already has an account at, the send should prefer that same-mint account — the mint can often route the payment internally with zero or minimal fees.

Many Cashu mints embed a fixed string in the BOLT11 `description` field of the invoices they generate (e.g. `"Minibits"`). When we recognize that string, we can infer which mint created the invoice and, if the user has a matching BTC account at that mint with sufficient balance, preselect it as the source.

## Goals

1. When a scanned or pasted BOLT11's `description` matches a configured mint, preselect the user's BTC cashu account at that mint as the send source.
2. Apply the same selection rule at all three BOLT11 entry points: `/scan` → `/send` navigation, `/send` `clientLoader` (URL hash / deep link), and `selectDestination` inside the send store (manual paste).
3. Keep the rule a pure, testable function with no hook dependencies.
4. Fail safe — if anything in the lookup breaks, fall back silently to the default account.
5. Validate the env var configuration at build time so misconfiguration can't reach production.

## Non-goals

- LN-address smart selection. LN addresses resolve asynchronously; applying the same rule there is a future extension.
- USD accounts. Scope restricted to BTC cashu accounts for this pass — USD→BTC conversion via NUT-05 adds exchange-rate complexity.
- Adding/removing mints based on scanned invoices. If the user doesn't have an account at the matched mint, we fall back to the default — we never prompt or silently onboard a mint.
- Runtime-editable configuration (no DB table, no admin UI). The data is essentially static.
- Dynamic detection from `/v1/info` mint name. Considered and rejected for this pass — explicit env config is simpler and deterministic.

## Rule

For a classified BOLT11:

```
if description is set
   and description ∈ configured map
   and user has a BTC cashu account at the mapped mintUrl
   and invoice has a non-null amount
   and account balance ≥ invoice amount
then   use that account as the send source
else   use the user's default account
```

Zero-amount invoices never match — we can't balance-check. The user will pick a source manually on the send page.

Balance comparison uses invoice amount with **no fee-reserve buffer**. A small chance exists that the matched account has just enough to cover the invoice but not the melt fee — the existing send flow surfaces an error at confirm-time in that edge case. This is acceptable for an optimization path.

## Components

### 1. Config schema and env var

**New file:** `app/features/send/mint-description-config.ts`

```ts
import { z } from 'zod';

export const MintDescriptionMapSchema = z.record(z.string(), z.string().url());
export type MintDescriptionMap = z.infer<typeof MintDescriptionMapSchema>;

// Used by vite.config.ts to validate the raw env var string at build time.
export const JsonMintDescriptionMapSchema = z
  .string()
  .transform((s) => JSON.parse(s))
  .pipe(MintDescriptionMapSchema);
```

**Env var:** `VITE_MINT_DESCRIPTION_MAP` — JSON string mapping description → mintUrl, e.g.

```json
{"Minibits":"https://mint.minibits.cash/Bitcoin"}
```

Keys are exact BOLT11 description strings. Values are mint URLs (must be valid URLs per schema).

**Build-time validation:** `vite.config.ts#validateEnv()` imports `JsonMintDescriptionMapSchema` and parses `VITE_MINT_DESCRIPTION_MAP` when present, mirroring the existing `VITE_GIFT_CARDS` pattern. Malformed JSON or a value that isn't a URL fails the build.

### 2. Pure selector

**New file:** `app/features/send/smart-source-selection.ts`

```ts
type SmartSelectionInput = {
  decoded: DecodedBolt11;
  accounts: ExtendedAccount<'cashu'>[]; // caller passes BTC cashu accounts
  defaultAccount: Account;
};

function selectSourceAccountForBolt11(
  input: SmartSelectionInput,
): Account;
```

Parses `VITE_MINT_DESCRIPTION_MAP` once at module load (safe to throw; build pre-validated). Exposes the selection logic as a pure synchronous function.

**Decision branches** (evaluated top to bottom; first mismatch returns `defaultAccount`):

1. `decoded.description` is set
2. Lookup in the parsed map returns a `mintUrl`
3. `accounts.find(a => a.mintUrl === mintUrl)` returns a match
4. `decoded.amount` is non-null
5. Matched account's balance ≥ `decoded.amount`
6. Return the matched account

### 3. Call sites

All three sites pass the pre-filtered BTC cashu account list and the user's default account into `selectSourceAccountForBolt11`.

**a) `/send` `clientLoader`** (`app/routes/_protected.send.tsx`)

After `classifyInput`, for `type === 'bolt11'`, read the accounts cache directly (loader has no hooks). Call the selector. Return `{ initialDestination, initialAccountId }` from the loader. `SendLayout` passes `initialAccountId` to the send provider — overrides the `accountId` query param for this flow.

**b) `/scan` navigate handler** (`app/routes/_protected.scan.tsx`)

For `type === 'bolt11'` after `validateBolt11`, read the accounts cache, call the selector, append `?accountId=<selected>` to the `/send` URL before navigating. The `/send` loader then sees the explicit id and uses it.

Alternative: skip this wiring on the scan side and let the loader handle it uniformly. Pick whichever keeps the diff smallest — functionally equivalent because the loader already does lookup on the hash.

**c) `selectDestination` in `send-store`** (`app/features/send/send-store.ts`)

In the BOLT11 branch, after validation, call the selector with `deps.getAccounts()` + `deps.getDefaultAccount()`. If the selected account differs from current `state.accountId`, update it.

### 4. Tests

**New:** `app/features/send/smart-source-selection.test.ts` — covers the 6 decision branches with fabricated inputs. Pure function, no mocks.

## Edge cases and failure modes

| Case | Behavior |
|---|---|
| Env var missing or empty | Empty map, selector always returns default. No warning. |
| Env var malformed | Build fails. Cannot ship misconfigured. |
| Invoice has no description | Default account. |
| Description doesn't match any mapped key | Default account. |
| Matched mint but user has no BTC account at it | Default account. |
| Matched account exists but balance < invoice amount | Default account. |
| Zero-amount invoice | Default account (can't balance-check). |
| Matched account balance covers invoice but not fee reserve | Selected anyway; error surfaces at melt-quote confirmation (existing path). |
| Two accounts at the same mintUrl (shouldn't happen for BTC, but) | First match — deterministic, no UX impact given currency narrows to BTC. |

No toasts, no warnings to the user. Smart selection is an optimization — invisible when it works, indistinguishable from current flow when it doesn't.

## Out of scope / future

- LN-address smart selection (requires async LNURL resolution before selection).
- USD account selection for BOLT11 (requires exchange-rate awareness and NUT-05 USD→BTC path).
- Auto-detecting mint identity from `/v1/info` (eliminates config but adds async fetching per scan).
- Admin-managed / DB-backed mint description table.
